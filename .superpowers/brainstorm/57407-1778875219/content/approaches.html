3 Approaches to Evolving the App

How do we layer in teacher features + AI without breaking what's there?

A

UX-first Redesign

Redesign navigation and information architecture first to accommodate roles + AI, then build features into the new shell. Student and teacher views diverge meaningfully from the nav level.

Student nav vs Teacher nav
Student
🏠 Today
✅ Tasks
📚 Courses
📝 Notes
📅 Timeline
🤖 AI Copilot
Teacher
🏫 My Classes
📋 Assignments
📊 Evaluations
📦 Materials
👥 Students
🤖 AI Insights

Pros

  • Coherent feel from day one
  • Teacher UX designed intentionally

Cons

  • More upfront design work
  • Need to refactor nav before adding features
B

Feature-first Expansion

Add teacher portal features (course management, study materials, grading) to the existing structure first. Redesign UX after the features are solid. AI layer comes last.

📝 Notes
📅 Timeline
+ Teaching (if teacher)
+ Evaluations (if teacher)
+ AI Copilot (later)

Pros

  • Fastest path to working teacher features
  • Low risk to existing student UX

Cons

  • Nav gets crowded fast
  • Feels "bolted on" before UX pass
C

AI-backbone Architecture Recommended

Design the AI copilot as the connective tissue of the app. All features — tasks, courses, teacher assignments — feed data into the AI layer. UX redesign and teacher features ship together, unified by the AI thread.

One nav, AI woven throughout
🏠 Today ← AI-curated
✅ Tasks + AI priority
📚 Courses + teacher assigns
📝 Notes + AI summary
📅 Timeline + AI schedule
🎓 Teaching teacher-only
🤖 AI Copilot floats as a persistent sidebar/panel across all sections

Pros

  • Feels like one coherent product
  • AI is a feature, not an afterthought
  • Teacher section is clean and intentional

Cons

  • Most design work upfront
  • AI integration needs Claude API
